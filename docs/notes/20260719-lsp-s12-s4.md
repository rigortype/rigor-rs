## ProjectContext + generation + epoch design

- **`ProjectContext`** (renamed from S3's `Analysis`, promoted to tier 1):
  `{ generation: u64, index: CoreIndex, disable: SuppressSet, folder:
  Option<Arc<SidecarFolder>> }` behind an `Arc`. Built once at startup, then
  **loop-owned** (in `Session`, not the immutable `ServerContext`) so `invalidate`
  can swap it. Workers capture an `Arc` clone at dispatch (the S3 seam); a worker
  holding the OLD `Arc` finishes against it and is generation-dropped.
- **`invalidate`** bumps `generation`, rebuilds `CoreIndex::for_project(...)` +
  `disable` from the retained config, and **preserves the sidecar** by cloning its
  `Arc` into the new context (the Ruby VM is NOT respawned). The config file is not
  re-parsed (matching the reference `ProjectContext#invalidate!`, which rebuilds
  from the same `@configuration`); the rebuild re-reads the signature dirs from
  disk, so changed `sig/**/*.rbs` content is picked up. (Structural `.rigor.yml`
  re-parse — new plugins/dirs — is a future enhancement; the same limitation the
  reference carries.)
- **`Session`** bundles the loop-owned mutable state (`buffers`, `debouncer`,
  `in_flight`, `epochs`, `project`, `cfg`, `results_tx`) so the lifecycle functions
  take `&mut Session` instead of a 10-parameter list (which would trip clippy's
  `too_many_arguments`). Query handlers (`hover`/`completion`/`document_symbols`)
  take `&ProjectContext`/`&BufferTable` directly and stay synchronous on the loop
  thread.
- **Three-axis stale-drop** in `handle_result`: a result publishes only if
  **version** (no edit past what was analysed, S3) AND **generation** (no
  `invalidate` since dispatch, S4) AND **epoch** (no close/reopen cycle since
  dispatch, S4) all match the current state. Any mismatch → DROP + `request_dispatch`
  the latest content under the current context (no-lost-update: the final state is
  always eventually published). A closed buffer drops silently.

## Synchronous-rebuild decision (orchestrator, overrides the plan's lazy rebuild)

The plan (item 7) specified a lazy rebuild on the next worker. The orchestrator
overrode this: **`invalidate` rebuilds SYNCHRONOUSLY on the loop thread.**
Rationale — invalidation events are RARE (config / `Gemfile.lock` / signature save),
unlike keystrokes, so paying a ~100-300 ms `CoreIndex::for_project` build inline is
acceptable UX, and it avoids a second concurrency hazard (a worker-produced context
swap racing the loop's reads). A code comment records the future optimization (lazy
async rebuild serving the old context until the stamped replacement lands) for if
profiling ever shows the stall matters.

## Invalidation trigger set

- **`workspace/didChangeWatchedFiles`** → invalidate + re-analyse ALL open buffers
  **only if** a changed URI is on the config + project-signature surface:
  `.rigor.yml`, `Gemfile.lock`, a project `*.rb`, or a `sig/**/*.rbs` (project sig
  dirs matter for rigor-rs per ADR-0033). An unrelated path (a `.txt`, a build
  artifact) does NOT invalidate — no needless rebuild. A watched `.rb` that is also
  an open buffer: the rebuild refreshes the project index and the open buffer
  re-dispatches (the cross-file *benefit* lands in S4b; S4 delivers the plumbing).
- **`workspace/didChangeConfiguration`** → always invalidate + re-analyse open
  buffers (payload ignored, v1).
- **Buffer `textDocument/didChange`** → NEVER invalidates (asserted explicitly in
  code + test): buffer edits are virtual, single-file scope; only the config /
  watched-file surface bumps the generation.

Re-analysing "all open buffers" routes each open URI through `request_dispatch`; a
URI with an old-generation worker still in flight is a no-op there and is covered by
the generation-drop re-dispatch, so the new context is always eventually applied.

## Dynamic registration

The client's `initialize` params are now threaded out of `Connection::initialize`'s
return (pre-S4 discarded):
`capabilities.workspace.didChangeWatchedFiles.dynamicRegistration`. When `true`, the
server sends one `client/registerCapability` request for the watcher globs
(`**/*.rb`, `**/.rigor.yml`, `**/Gemfile.lock`, `**/sig/**/*.rbs`); when
absent/false it degrades gracefully — no registration, and the server still honours
any `didChangeWatchedFiles` the client sends statically (no regression).

**Gotcha:** the `initialized` notification is CONSUMED by the `lsp-server`
handshake (`initialize_finish` waits for it), so it never reaches the message loop.
The registration is therefore sent once at the **top of `main_loop`** (which holds
the connection + the capability flag), not from a notification handler.

## How the S3 close+reopen version-reuse nit is closed

A close+reopen that reuses the LSP version (VS Code resends version 1 on reopen)
could let a stale pre-close worker's result publish for the new buffer: version
matches, and generation is project-scoped so it does NOT bump on reopen. The fix is
a **per-URI open-epoch** (`Session.epochs`): a monotonic counter bumped on every
`didOpen` AND `didClose`, persisting across close (unlike the buffer entry). A
worker stamps its result with the epoch at dispatch; a result whose epoch no longer
matches is dropped + re-dispatched. Interleaving:

1. `didOpen` v1 (epoch 1) → worker A captured epoch 1, held in flight.
2. `didClose` (epoch 2) — clears markers; A still in flight.
3. `didOpen` v1 reopen, new content (epoch 3) — the reopen's dispatch no-ops (A
   still in flight).
4. A returns: version matches (1) and generation matches, but epoch 1 ≠ current 3 →
   DROPPED + re-dispatch → the reopened content is analysed fresh and published.

A dedicated test drives exactly this interleaving and asserts the reopened (clean)
content publishes, not the stale pre-close (typo) result — the S3 behaviour would
have published the typo.

## Non-flaky test approach + 5× result

All S4 concurrency tests are DETERMINISTIC — no wall-clock races. They reuse the S3
**worker-gate seam** (extended to `Fn(i32, u64)` = version + generation, so a test
can hold/panic a worker keyed on either axis) and **hover round-trips as FIFO
synchronisation barriers**, with a 30 s debounce where a stray timer would otherwise
interfere.

- `integration_s4_generation_stale_drop_after_invalidate` — a **recording** gate
  holds the gen-0 worker and records every `(version, generation)` it gates; an
  invalidation bumps the generation; on release the gen-0 result is dropped and a
  gen-1 worker re-runs. The proof is the recorded gen-1 call: without the generation
  guard the gen-0 result would publish directly and no gen-1 worker would run
  (publish count alone can't distinguish — the content is identical). Keying the
  hold on generation lets the same-version re-dispatch run freely.
- `integration_s4_watched_files_relevant_reanalyzes_all_open_buffers` /
  `..._unrelated_does_not_invalidate` — a `.rigor.yml` change re-publishes BOTH open
  buffers; a `.txt` change re-publishes neither (hover barrier proves no publish).
- `integration_s4_did_change_configuration_reanalyzes` — invalidate + re-publish.
- `integration_s4_buffer_didchange_never_invalidates` — an edit to `a.rb`
  re-publishes ONLY `a.rb`; `b.rb` stays quiet (an invalidate would re-publish both).
- `integration_s4_dynamic_registration_sent_when_advertised` /
  `..._not_advertised_but_watched_files_still_honored` — asserts the
  `client/registerCapability` request is/ isn't sent, and that watched-files are
  honoured either way.
- `integration_s4_close_reopen_version_reuse_drops_stale_preclose_worker` — the nit
  interleaving above (two same-version release tokens: one for the pre-close worker,
  one buffered for the epoch-drop re-dispatch).
- `watched_file_relevance_matches_the_config_and_signature_surface` — pure unit test
  of the path-matcher.

**5× flake check:** the lsp test binary (`lsp::`, 43 tests) ran **43/43 five times,
0 failures**, and **43/43 once with `RAYON_NUM_THREADS=1`** (the worst case — a held
worker occupies the only pool thread). Safe by construction: every test's
release/shutdown path uses only the LOOP thread, and the epoch-drop / generation-drop
re-dispatches free the held pool thread before the next worker needs it, so no test's
progress depends on another's worker running — no cycle.
