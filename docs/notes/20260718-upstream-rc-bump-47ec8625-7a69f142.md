## Slice 1 — `suppression.unknown-marker` (new rule)

Upstream `4e0ca475`, itself authored **from this port's harness feedback**
(`docs/notes/20260716-upstream-feedback.md` item 2). Fires on a `rigor:`-prefixed
marker OUTSIDE the suppression grammar that reads like an attempted suppression —
the RuboCop reflex `# rigor:disable-next-line <rule>` / `# rigor:enable <rule>` —
so it silently suppresses nothing. Ported into `suppression_marker_diagnostics`:
the bare-marker path now falls through to an unknown-marker pass
(`UNKNOWN_SUPPRESSION_MARKER`), matching the reference's BARE→UNKNOWN chaining
with the same prose escape. Message byte-exact. Wired the full surface (rule id,
catalogue/`explain` entry, `IMPLEMENTED_RULES`, `ALL_CANONICAL_RULES` so the
config audit stops flagging the now-known id, warning across all profiles).

## Slice 2 — Kernel intrinsic explicit-`Kernel.`-receiver fold

Upstream `c9d2e473` (also from this port's feedback, item 1): `module_function`
exposes each Kernel intrinsic as a public singleton on the Kernel module object,
so `Kernel.p(x)` / `Kernel.format(...)` / `Kernel.String(x)` dispatch to the SAME
fold as the implicit-self spelling. rigor-rs folded only the `receiver: None`
path; `type_call` now routes a receiver typing to `Singleton[Kernel]` into the
shared `type_implicit_self_call` fold. Typed on the receiver (not node-spelled),
so a namespaced user `Kernel` constant — which types Dynamic — cannot slip
through. The reference's FP-reducing half (a user-redefined `obj.format` must not
be hijacked) was **already** satisfied: rigor-rs's format fold is implicit-self +
`file_defines_method`-guarded, so it never hijacked an explicit foreign receiver.
