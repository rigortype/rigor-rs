# ADR-0038 / Slice 1 audit — concerns before implementation (2026-07-06)

> **Status: absorbed.** ADR-0038 was revised the same day to make the
> decision-level concerns normative — §3 (block-scope semantics: entry-widen,
> fact locality, shadow-clearing, descend-XOR-widen, per-rule state), the
> Slice 1 guards (`arr[Range]` scope limits + recorded debt, `type_dot_new`
> leniency invariant, top-level parity confirmation), the gate's matched
> non-regression criterion, and the pinned-oracle caveat. This note remains as
> the review record and the rationale trail.

An independent review of the [ADR-0038](../adr/0038-flow-substrate-incremental-narrowing.md)
decision and the CURRENT_WORK Slice 1 plan (foundation + possible-nil in block
scopes), read against the current `flow_eval` implementation
([`crates/rigor-infer/src/lib.rs`](../../crates/rigor-infer/src/lib.rs), `always_truthy_snapshots` /
`flow_eval_stmt` / `flow_eval_scope`, lines ~560–735) and the current
`check_nil_receiver` path ([`crates/rigor-rules/src/lib.rs`](../../crates/rigor-rules/src/lib.rs),
`nilable_local_core_arm` / `nil_local_is_guarded`, lines ~1259–1500).

**Verdict:** the design skeleton is sound — unifying both rules onto the threaded
`flow_eval`, the "unmodeled construct ⇒ decline" backstop, and the measured
gap-reduction gate all match the recon lesson. The concerns below are about the
Slice 1 *specification* being thin exactly where the recon flagged the work as
"FP-delicate" (possible-nil across block / top-level scopes). Ranked by severity.

## 1. The "straight-line within the block" assumption collides with 0-or-many execution (top FP risk)

The plan says "straight-line within the block; widen block-captured writes on
exit." Widening only *on exit* has two holes:

- **Multiple execution (loop-shaped).** In
  `x = arr[0]; list.each { x.frob; x = nil_source }` a read inside the block that
  lexically precedes the block's own capture-write still, on the 2nd iteration,
  observes the post-write value. This is why the current code widens loops
  wholesale in the `other` arm. Descending into a block therefore requires
  widening every capture-write in that block's span **on entry** (apply
  `widen_flow_writes` to the block body span *before* descending), not just on
  exit.
- **Deferred / stored blocks.** `callback = obj.on_event { x.size }` — the block
  runs later, so its lexical position carries no fact. The entry-widen above keeps
  the nilable side sound, but the robust framing for Slice 1 is to **only admit
  facts whose source AND use are in the same block body** (the treemaps target
  pattern), and decline anything crossing a block boundary.

**Recommendation:** make Slice 1's spec explicit — "fact source and use within the
same block body; widen the whole descended block span on entry."

## 2. Block-parameter shadowing

A `HashMap<String, …>` keyed by local name leaks facts across a same-named block
parameter: `x = arr[0..n]; list.each { |x| x.size }` is a *different* `x`, but the
outer fact would fire ⇒ FP. Slice 1 must **clear every name bound by the block's
parameters on descent** — including numbered params (`_1`…) and `it`. The
nilability map needs the same fresh-scope treatment that `flow_eval_stmt` already
gives `Definition`/`ClassDef`/`ModuleDef` for the constant env.

## 3. `in_loop_or_block` flag splits meaning across the two rules

Today `always_truthy_snapshots` uses `in_loop_or_block` to suppress recording
inside blocks/loops. Once blocks are descended for nilability, that single flag
means two things — "suppress always-truthy" but "record nilability." Reusing one
flag risks either accidentally enabling always-truthy inside blocks or suppressing
the nilability Slice 1 depends on. **Keep per-rule flags / state** distinct from
the start.

## 4. Retiring the span-scan must not regress existing matched coverage

Slice 1 rewires `check_nil_receiver` to fire from a new
`nilable_receiver_snapshot` and retires the `enclosing_def` span-scan. The current
span-scan path (with `nil_local_is_guarded`) already fires on real def-scope cases
(the two byte-exact possible-nil fixtures + corpus matched). "Ship nothing that
closes no gaps" is gated, but **non-regression of existing matched** is a separate
property — add corpus matched-count non-regression (not just the fixtures) to the
Slice 1 gate, so a stricter new path silently dropping a currently-firing case is
caught.

## 5. Hand-recognizing `arr[Range]` overload selection is a divergence point

Recognizing "the single-Range-arg form" partially reimplements the reference's
overload resolution. The firing condition ("method absent on NilClass, present on
the non-nil arm") means a wrong arm-class lands on the **wrong-fire** side. Guard
it tightly:
- apply **only** when the receiver is a confirmed core `Array`/`String` (a
  user-defined `[]` ⇒ decline);
- unit-test the arm mapping (`arr[Range]`→`Array`, `str[Range]`→`String`) against
  the oracle;
- explicitly declare out-of-scope-for-Slice-1 (⇒ decline): endless ranges, the
  two-arg `arr[i, len]` form, `str[Regexp]`, etc.

Record this as debt: ADR-0038's "no divergence" goal ultimately wants the
reference's real overload-selection ported here, not a hand-rolled special case.

## 6. Shared `type_dot_new` must carry the 2026-06-26 leniency invariant

Extracting a shared `type_dot_new` for `X.new(…){…}` must preserve the
CURRENT_WORK correctness finding: **non-core `.new` instances are typed but NOT
witnessed.** If the witnessing gate (core-RBS-known) doesn't follow the block path
out of the plain path's inline implementation, the `Struct.new(...).new` /
block-form FP class re-appears. Be explicit about where the witnessing gate lives
after extraction.

## 7. Smaller points

- **Top-level fire parity.** After the rewire the `enclosing_def` gate is gone, so
  bare top-level `x = arr[0..n]; x.size` becomes fireable. Confirm E2E that the
  reference actually fires at top level before enabling it; if not, keep a
  scope condition matching the reference.
- **Descent vs widen must be mutually exclusive.** Once a Call-with-block is
  descended, do NOT also run it through the `other`-arm span-widen — else the
  descended facts are immediately widened away (silent no-op bug).
- **Monotonic gap-closure assumes a pinned oracle.** ADR-0038's "100% parity"
  can move when the upstream pin bumps (v0.2.7 → future); the monotonicity claim
  holds only against a fixed oracle.
- **Fixed (this pass):** CURRENT_WORK.md Slice 2+ line used `‖` (U+2016) for the
  `||` operator — corrected to `&&/||`.

## Bottom line

The backstop + measured-gate skeleton is correct. The one substantive gap is that
Slice 1's write-up is light on block execution semantics (0/many/deferred) and
parameter shadowing — precisely the "FP-delicate" area the recon warned about.
Before coding, pin Slice 1's spec to: *fact source/use within the same block body
+ entry-widen the descended block span + clear block-param names.*
