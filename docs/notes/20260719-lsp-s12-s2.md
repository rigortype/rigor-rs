# LSP §12 S2 — per-URI 200 ms `didChange` debounce (2026-07-19)

Implements slice **S2** of the [two-tier plan](20260719-lsp-s12-two-tier-impl-plan.md)
([ADR-0029](../adr/0029-lsp-architecture.md) §debounce), built directly on
[S1](20260719-lsp-s12-s1.md). Blast radius: `crates/rigor-cli/src/lsp.rs`
(+ tests) only. This slice INTRODUCES an intentional observable change:
`didChange` no longer publishes immediately — it publishes 200 ms after the last
change for that URI. The diagnostic CONTENT is unchanged from S1; only WHEN a
`didChange` publish fires changes.

## Semantics (matching the reference `DiagnosticPublisher` `debounce_seconds: 0.2`)

- **didOpen** → publishes IMMEDIATELY (fast first paint), NOT debounced — as S1.
  (The reference routes `didOpen` through `publish_for` too, i.e. debounced; the
  binding rigor-rs plan §4 and the S2 spec deliberately keep `didOpen` immediate
  for first-paint latency. Faithful to the plan, a documented divergence from the
  reference's uniform `publish_for`.)
- **didChange** → updates the BufferTable SYNCHRONOUSLY (so hover/completion/
  documentSymbol see the latest text at once), then schedules a publish
  `ctx.debounce` after this change. A further `didChange` within the window
  RESETS the deadline — a burst coalesces into ONE publish of the FINAL content.
  Only the PUBLISH is deferred.
- **didClose** → CANCELS any pending debounce for the URI, then clears
  diagnostics (empty publish). No stale publish fires after a close.
- **hover / completion / documentSymbol** → answered SYNCHRONOUSLY from the
  current buffer, never interacting with the debounce.

## Debounce structure

- **`Debouncer`** (new): a `HashMap<String, Pending { uri, deadline: Instant }>`,
  owned solely by the loop thread. `schedule(uri, deadline)` inserts/replaces
  (the coalescing rule — last change in a burst wins the deadline, at most one
  pending publish per URI); `cancel(uri)` drops a pending publish (didClose);
  `earliest() -> Option<Instant>` is the min deadline; `take_due(now) -> Vec<Uri>`
  removes and returns every URI whose deadline has passed. **The struct holds no
  clock** — the caller computes deadlines (`Instant::now() + interval`) and passes
  `now` to `take_due`, so the fire/no-fire decision is a pure function of explicit
  `Instant`s.
- **`select!` timeout arm.** The loop computes `debouncer.earliest()`:
  `Some(deadline)` ⇒ block with `default(deadline.saturating_duration_since(now))`
  (clamped to 0 if already passed); `None` ⇒ block with no timeout. On timeout,
  `fire_due` calls `take_due(now)` and, for each due URI, re-dispatches from the
  **latest** buffer content via `dispatch_diagnostics` (so a coalesced burst
  analyses the final text). The dispatched `Computed` is published by the existing
  S1 top-of-loop drain — the single-writer invariant is preserved (only the loop
  thread sends `publishDiagnostics`). An incoming message wakes `select!`
  immediately regardless of the timeout, so `didClose`'s cancel is serviced
  without waiting out the deadline.
- **Injectable interval.** `ServerContext.debounce: Duration`, production =
  `DEBOUNCE_DEFAULT` (200 ms) at the single construction site in `run_stdio`.
  Never hardcoded at a use-site.

## Non-flaky test approach + 3× flake result

The timing seam is the clockless `Debouncer`: the coalescing/cancel/earliest
invariants are proven in **pure deterministic unit tests** driven by explicit
`Instant` values (`debouncer_coalesces_and_last_deadline_wins`,
`debouncer_cancel_drops_pending`, `debouncer_earliest_is_the_min_across_uris`) —
no thread, no sleep, no wall-clock dependency. The strict last-writer-wins /
exactly-one-due behavior is therefore established without any race.

The loop-wiring integration tests (real `main_loop` over `Connection::memory()`)
each pick an interval that its assertions cannot race:

- `integration_didchange_deferred_until_interval` — interval 150 ms; asserts NO
  publish arrives in a 20 ms window (20 ms ≪ 150 ms deadline), then that it does.
- `integration_rapid_didchanges_coalesce_to_one_publish` — interval 120 ms; both
  didChanges are queued to the connection before the deadline can elapse, so the
  loop processes schedule-then-reschedule microseconds apart and fires ONCE with
  the final content; a follow-up hover round-trip proves no second publish. (The
  strict invariant is the deterministic unit test above; this is a behavioral
  belt-and-suspenders with a generous margin.)
- `integration_didclose_cancels_pending_no_stale_publish` and
  `integration_hover_during_debounce_window_sees_latest_text_no_publish` —
  interval 30 s, so the deferred publish cannot fire during a millisecond-scale
  test; a hover round-trip asserts the next message is a Response, never a stale
  publish, and (for hover) that it reflects the edited buffer text.
- `integration_didchange_to_clean_republishes_empty` (S1 test, ADAPTED) — S1
  assumed an immediate didChange publish; now uses a 10 ms interval and asserts
  the deferred publish arrives and is empty. Assertions preserved, not deleted.

**3× flake check:** `cargo test -p rigor-cli lsp::` run 3 additional times after
the initial pass — 30/30 each run, 0 failures. Full workspace `cargo test`:
all suites green.

## Seam S3 depends on

- **Where compute still moves off-thread is unchanged from S1:**
  `dispatch_diagnostics` (still inline in S2) is the rayon-spawn swap point; both
  the `didOpen`-immediate path and the `fire_due` debounce-fire path route
  through it, so S3 gets rayon dispatch for both by changing one body.
- **`fire_due` reads the latest buffer via `BufferTable::snapshot(uri) ->
  (text, version)`** — S3's version stale-drop compares this `version` at
  `publish_result` time.
- **The `Debouncer` is loop-owned state** alongside the BufferTable; S3's
  "at most one in-flight dispatch per URI, edits during flight reset the deadline"
  builds on the same pending map (a re-`schedule` after a stale drop).
