# LSP §12 S3 — rayon dispatch + version stale-drop + shared sidecar (2026-07-19)

Implements slice **S3** of the [two-tier plan](20260719-lsp-s12-two-tier-impl-plan.md)
([ADR-0029](../adr/0029-lsp-architecture.md) §worker-pool), built on
[S1](20260719-lsp-s12-s1.md) (BufferTable + `select!` + worker-results channel)
and [S2](20260719-lsp-s12-s2.md) (per-URI 200 ms debounce). Blast radius:
`crates/rigor-cli/src/lsp.rs` (+ tests) only. No `rigor_infer`/`rigor_rules`/
`rigor_parse` edits; the sidecar/`RubyFolder` sharing reuses the `check`
pipeline's existing public API.

**Scope (orchestrator decision):** rayon dispatch + shared sidecar + **VERSION**
stale-drop only. **GENERATION** stale-drop is DEFERRED to S4 (it lands with its
trigger — `ProjectContext` invalidation). No generation field in S3.

Diagnostic CONTENT is byte-for-byte unchanged (same `compute_diagnostics`):
harness live + snapshot both 216/218, 0 unregistered FP — identical to pre-S3.

## What changed

- **Dispatch → `rayon::spawn`.** The old inline `dispatch_diagnostics` (compute on
  the loop thread) is replaced by `spawn_worker`: it captures a buffer snapshot
  `(text: String, version)`, an `Arc<Analysis>` clone (the shared analysis
  context), a `worker_gate` clone (test seam), and a `results_tx` clone, then
  `rayon::spawn`s the EXACT `check` compute off-thread and sends one `Computed`.
  Both the didOpen-immediate path and the debounce `fire_due` path now route
  through `request_dispatch` → `spawn_worker`. The `select!` (b) `results_rx` arm —
  dormant since S1 — is now genuinely fed by async worker pushes.
- **`ServerContext` split.** The analysis-relevant state (`index: CoreIndex`,
  `disable: SuppressSet`, `folder: Option<SidecarFolder>`) moved into a new
  `Analysis` struct held behind `Arc` in `ServerContext`, so a clone is captured
  per worker. `compute_diagnostics(&Analysis, &str)`; hover/completion read
  `ctx.analysis.index` (still synchronous on the loop thread — they do NOT go
  through rayon).
- **Pre-warm at initialize.** `run_stdio` calls `rayon::spawn(|| {})` once after
  building the context, so the global pool spawns its threads at startup rather
  than on the first keystroke. Pool size honours `RAYON_NUM_THREADS` natively; no
  `--workers` LSP flag added.
- **VERSION stale-drop** in `handle_result` (see lifecycle below).

## The one-in-flight / no-lost-update lifecycle (the core of S3)

Loop-owned state (single-threaded, alongside the BufferTable + Debouncer):
`in_flight: HashSet<String>` — the URI keys with a rayon worker running. The
buffer's `version` is the single source of truth for staleness; no separate
"published version" mark is needed.

**`request_dispatch(uri)`** — the one-in-flight gate:
- if `in_flight` contains `uri` → **do nothing** (a worker is already running; its
  result will publish if current, or trigger a re-dispatch if stale);
- else read the LATEST snapshot and `spawn_worker` (closed/unknown buffer ⇒ skip).

**`spawn_worker(uri, text, version)`**:
- insert `uri` into `in_flight`;
- **cancel the URI's pending debounce** — the in-flight worker now covers the
  latest content, so the deferred publish is redundant (this is what makes the
  final publish happen *exactly once*, not once-per-trigger);
- `rayon::spawn` the worker (compute + send one `Computed`).

**`handle_result(computed)`** — the loop's single-writer publish point:
1. `in_flight.remove(uri)` — a returned worker is no longer in flight;
2. match `buffers.current_version(uri)`:
   - `None` (closed) → **DROP** (a `didClose` already cleared its markers);
   - `Some(cur)` and `cur == computed.version` → **PUBLISH** (result is live);
   - `Some(_)` (buffer moved past `version`, a newer edit superseded it) → **DROP**
     and `request_dispatch(uri)`. Because `in_flight` was just cleared, this
     spawns a fresh worker for the newest snapshot — so the final buffer state is
     ALWAYS eventually published, and a dropped stale result never strands the
     latest content unpublished (**no lost update**).

**Event wiring:**
- `didOpen` → record buffer, cancel debounce, `request_dispatch` (immediate spawn
  if idle — fast first paint).
- `didChange` → update buffer synchronously (hover/completion see latest at once),
  `debouncer.schedule(now + interval)`. **No dispatch** — an edit during flight
  ONLY updates the buffer + resets the deadline; it never spawns a second worker.
- `fire_due` → `request_dispatch` per due URI (skips if still in flight; the eventual
  stale-drop re-dispatch handles it).
- `didClose` → cancel debounce, clear markers; a worker still in flight is left to
  finish and its result is dropped (current version `None`).

**Invariants established:** at most one worker per URI (only `spawn_worker` spawns,
only under `!in_flight`, all on the single loop thread); every worker sends exactly
one `Computed` so `in_flight` always clears; only `version == current` publishes so
no stale/intermediate version ever escapes; and the newest content is always
eventually published via the stale-drop re-dispatch.

## Shared sidecar under concurrency (confirmed)

The single `SidecarFolder` is shared as `&(dyn RubyFolder + Sync)` inside each
worker's `compute_diagnostics` — the EXACT pattern the `check` pipeline uses
(`main.rs:243-244`, shared across `par_iter` workers). Here it is reached via the
`Arc<Analysis>` capture; the folder's internal `Mutex` serializes folds across
concurrent LSP workers (contention accepted; measure later per ADR-0029). The
build confirms `Arc<Analysis>: Send + Sync` (⇒ `CoreIndex`, `SuppressSet`,
`SidecarFolder` are all `Send + Sync`) — no `pub`/`Sync` change was needed; the
`check` pipeline already relies on the same bounds.

## Panic / never-stuck

The worker body is `catch_unwind`-wrapped *outside* the compute: a panic in the
gate or the compute yields an empty-diags result rather than a lost send, so the
worker ALWAYS sends exactly one `Computed` and the URI's `in_flight` always clears
(`compute_diagnostics` is itself panic-isolated per ADR-0016; the outer catch
backstops the seam + any unexpected panic). Because the catch is *inside* the
spawned closure, no panic ever escapes to rayon (no abort).

## Shutdown

On `shutdown`/`exit` the loop returns; `results_tx`/`results_rx` drop. A detached
worker's later `send` on the unbounded channel returns `Err` (ignored) — never
blocks. `run_stdio` drops the connection and joins the IO threads (the loop
thread), not the rayon workers, so shutdown returns promptly regardless of a
worker still running. Verified deadlock-free even with a held worker (test below).

## Non-flaky test approach + 5× flake result

Real rayon makes timing nondeterministic, so the concurrency tests drive the
lifecycle DETERMINISTICALLY via a **worker-gate seam** (`ServerContext.worker_gate:
Arc<dyn Fn(i32) + Send + Sync>`, production = no-op): a test gate blocks a worker
at a target version until released, or forces it to panic. Ordering is pinned by
the gate + **hover round-trips as synchronization barriers** (the loop services
messages FIFO, so a returned hover response proves every earlier message was
processed) — no test depends on a wall-clock window.

- `integration_s3_edit_during_flight_drops_stale_and_publishes_final_once` — hold
  v1 mid-flight, edit to v2, release v1; the stale v1 is dropped and v2 (the final
  content) publishes exactly once. 30 s debounce so ONLY the stale-drop re-dispatch
  (never the clock) drives the publish.
- `integration_s3_burst_edits_coalesce_to_final_no_stale_publish` — hold v1, burst
  v2..v5 (typos) + v6 (clean); one-in-flight means v2..v5 never spawn; only v6 is
  re-dispatched → exactly one publish of the final content, no stale version escapes.
- `integration_s3_worker_panic_does_not_stick_the_uri` — v1's worker panics (caught)
  → empty result published → a later edit is still analysed and published (not stuck).
- `integration_s3_shutdown_with_worker_in_flight_does_not_hang` — hold a worker,
  shut down; the loop returns without waiting; the worker is released post-shutdown
  (its send finds the rx gone). No hang.

**Deadlock-freedom under `RAYON_NUM_THREADS=1`** (the worst case — a held worker
occupies the only pool thread): safe by construction, because every test's
`release`/`shutdown` path uses only the LOOP thread (never the pool), so no test's
progress depends on another's worker running — no cycle. Verified: the 4 S3 tests
pass 3/3 with `RAYON_NUM_THREADS=1`.

**5× flake check:** `cargo test -p rigor-cli lsp::` — 34/34 each of 5 runs, 0
failures. Full workspace `cargo test`: all suites green.

## S4 attach points

- **Generation stale-drop** attaches to `Computed` (add a `generation: u64` field)
  + `handle_result` (publish only if BOTH `version` AND `generation` match) +
  `spawn_worker` (capture the current generation). The version-guard structure is
  already the template — generation is a second, orthogonal equality check.
- **`ProjectContext` swap** attaches to `Analysis` behind the `Arc`: S4 replaces
  the single `Arc<Analysis>` with a generation-stamped `Arc<ProjectContext>` that
  `invalidate!` (watched-files / config) bumps; the loop keeps serving hover from
  the old `Arc` until a stamped replacement lands, and workers capture whichever
  `Arc` was current at dispatch (a worker computing against a superseded context
  is dropped by the generation guard). The `Arc`-capture-per-worker seam is exactly
  what S3 built.
- **Dynamic registration** (`client/registerCapability` for watched-files globs)
  is new wiring in the `initialized` handler — independent of the S3 loop.
