# LSP §12 S1 — BufferTable + `select!` loop + worker-results channel (2026-07-19)

Implements slice **S1** of the [two-tier plan](20260719-lsp-s12-two-tier-impl-plan.md)
([ADR-0029](../adr/0029-lsp-architecture.md)). Pure refactor of
`crates/rigor-cli/src/lsp.rs`; observable behavior byte-identical. S2 (debounce),
S3 (rayon dispatch + stale-drop), and S4 (ProjectContext invalidation) are NOT
touched — S1 only lays their seams.

## What changed

- **BufferTable** replaces the raw `HashMap<String, String>`. Entries carry
  `{ bytes, version, dirty }` (ADR-0029). `didOpen` → `open` (dirty=false),
  `didChange` → `change` (dirty=true, version bumped), `didClose` → `close`.
  `text(&Uri) -> Option<&str>` is the accessor threaded through
  hover/completion/document_symbols. `uri_key` semantics unchanged. Nothing
  branches on `version`/`dirty` in S1 — they are maintained for the S2/S3
  consumers (debounce, temp-file `BufferBinding`, version stale-drop).
- **`select!` loop.** `for msg in &connection.receiver` → a
  `crossbeam_channel::select!` over (a) the connection receiver and (b) a new
  internal **worker-results** channel (`unbounded::<Computed>`). `didOpen`/
  `didChange` no longer publish inline; they *dispatch* via
  `dispatch_diagnostics` (S1: `compute_diagnostics` inline on the loop thread,
  then `results_tx.send`). The loop is the **sole** sender of
  `publishDiagnostics` (`publish_result`).
- **`crossbeam-channel`** added as a direct dep of `rigor-cli` (already an
  unpacked transitive dep of `lsp-server`; offline-safe, no fetch).

## Ordering: why it stays byte-identical

A bare `select!` over a synchronously-fed channel could reorder: after the
connection arm enqueues a result, `select!` may pick the next connection message
before the result is drained. To guarantee today's order, the loop **drains the
results channel (`try_recv`) at the top of every iteration, before `select!`
blocks.** So a dispatched compute is always published *before* the next
connection message is serviced — identical to the former inline `publish`. A
consequence: the (b) `select!` arm is **dormant in S1** (results are drained
before we block); it comes alive in S3 when rayon workers push results
asynchronously while the loop is blocked in `select!`. `didClose` still publishes
its empty set directly on the loop thread (an idle-clear, not a compute — it does
not go through the worker channel; S2's "didClose cancels pending + clears" builds
on this).

## Golden-sequence verification method

Before refactoring, a throwaway test drove the **pre-refactor** `main_loop` over
`Connection::memory()` and printed the exact emitted message sequence for
didOpen(typo)/didChange(clean)/didOpen(clean)/hover/didClose/shutdown. The
permanent integration tests (`integration_*` in `lsp.rs`) assert that captured
sequence against the refactored loop: one `call.undefined-method` at range
(1,2)–(1,8) sev Error src "rigor"; empty republish on clean didChange; empty on
didClose; hover Response id 2 carrying markdown containing `42`. All pass —
byte-identical.

## Seam decisions S2/S3 depend on

- **Where compute moves off-thread:** `dispatch_diagnostics` body. S3 swaps the
  inline compute for `rayon::spawn` capturing a buffer snapshot + `results_tx`
  clone; the loop's drain/publish is unchanged.
- **Where the version stale-drop lands:** `Computed.version` +
  `publish_result`. S3 drops a result if the buffer's version moved on.
- **The results channel is created inside `main_loop`** (not a param), since S1
  is its only feeder. S3 clones `results_tx` into worker closures.
- **Single-writer invariant:** only the loop thread sends `publishDiagnostics`
  (both the top-of-loop drain and the `didClose` direct clear run on it).
